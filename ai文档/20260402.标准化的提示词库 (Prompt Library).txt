关于”标准化的提示词库 (Prompt Library)“，能给我仔细地介绍一下吗

在管理一个跨行业、多 Agent（OpenClaw 机器人）的复杂系统时，标准化的提示词库 (Prompt Library) 不仅仅是一个“文档库”，它是这支机器人团队的**“员工手册”和“协作协议”**。
由于你的系统涉及医疗、酒店、零售等多个专业领域，如果提示词不标准，机器人可能会在处理“挂号”（医院）和“挂账”（酒店）时产生歧义。

1. 为什么要标准化？（解决“语境漂移”）
在一个跨行业系统中，同一个词在不同场景下含义完全不同：
    . “库存 (Inventory)”：在零售是指货架商品，在酒店是指客房数量，在医院可能是指床位或血库。
    . “用户 (User)”：在物业是业主，在医院是患者，在物流是收件人。
标准化的词库能确保所有 OpenClaw 机器人使用统一的逻辑基准，避免数据关联时出现灾难性错误。

2. 标准化提示词库的结构
一个成熟的提示词库通常由以下四个层级组成：
    A. 基础身份层 (System Role Layer)
    定义机器人的核心价值观和行为准则。
        示例： “你是一个跨行业系统的‘数据审计机器人’。你的核心准则是在处理医疗数据与零售数据关联时，必须优先遵守 HL7 医疗隐私协议。”

    B. 领域知识层 (Domain Knowledge Layer)
    为不同行业的机器人注入特定的业务规则。你可以为每个行业建立一个 Prompt 模块：
        . 酒店模块： 包含 PMS 术语、房态转换逻辑。
        . 物流模块： 包含电子面单格式、轨迹节点定义。
        . 医院模块： 包含分诊标准、医保结算规则。

    C. 任务模版层 (Task Template Layer)
    针对具体操作的标准化格式（如：提取、转换、代码生成）。
        . 数据转换 Prompt： “将[输入行业]的交易记录转换为[目标行业]的结算格式，保留 UID 映射。”
        . 错误处理 Prompt： “如果物流配送进入物业区域时道闸未开启，请调用物业机器人的 API 检查车辆授权状态。”

    D. 输出约束层 (Output Constraints Layer)
    规定机器人输出的具体格式（JSON、SQL、Markdown等），确保机器人之间的数据可以直接对接。

3. 如何构建和管理这个词库？
    第一步：建立“行业本体词典” (Ontology)
    在编写 Prompt 之前，先定义一套全局通用的术语。
        . 全局实体： Global_UID（人）, Global_AssetID（物）, Global_LocationID（空间）。
        . 让机器人学习这个词典，要求其在所有生成内容中强制使用这些 ID。

    第二步：采用“结构化 Prompt”编写法
    推荐使用类似 XML 或 Markdown 的结构来编写 Prompt，方便机器人解析。
        结构化示例：
            <Task> 跨行业积分兑换逻辑生成 </Task>
            <Context> 用户在商超消费产生的积分需要抵扣物业费 </Context>
            <Reference> 见《商超-物业结算矩阵.v1》 </Reference>
            <Constraint> 严禁在非加密环境下透传用户医疗病历数据 </Constraint>
            <Output_Format> JSON (Fields: UserID, DeductAmount, SuccessFlag) </Output_Format>

    第三步：版本控制 (Prompt Ops)
    像管理代码一样管理 Prompt：
        . 使用 Git 存储 Prompt 库。
        . 当你的“医院”业务流程更新时（例如增加了新的医保接口），同步更新相关的 Prompt 版本，并推送到所有关联的 OpenClaw 机器人。

4. 在 OpenClaw 团队中的应用流程
    1. 分发： 机器人启动时，从标准库加载对应的 System Prompt。
    2. 动态组合： 当需要处理“酒店住客在商超买药并由物流送往病房”这种复杂任务时，中央调度器会组合酒店、零售、物流和医院四个模块的提示词。
    3. 自动迭代： 通过 QA-Bot 监测输出结果。如果发现机器人在处理“停车场收费”时逻辑有误，直接在 Prompt Library 中修复，所有涉及停车业务的机器人都将即时更新。

建议：
    你可以先从一个**“跨行业实体映射表”**开始你的词库建设。比如，明确定义“在所有行业模块中，
    只要涉及人的唯一识别，必须统称为 Subject_ID”。这小小的一步，能省掉后期 80% 的调试时间。
