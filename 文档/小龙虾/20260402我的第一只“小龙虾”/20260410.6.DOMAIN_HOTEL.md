# Domain Knowledge: 现代旅店管理系统 (PMS) 与前台结算

> **[System Directive]**
> 此文件包含特定业务领域的硬性交互与视觉规范。在生成当前领域的 UI 组件时，本文件中的“视觉映射规则”具有最高执行优先级。

## 1. 业务全景与核心目标 (Business Context & Goals)
- **核心用户**：高频操作的前台接待员、大堂值班经理。
- **业务痛点**：传统 PMS（如 Oracle OPERA）界面极其臃肿，信息密度分布不合理。过载的无层级数据导致前台视线频繁跳跃，账单核对极其耗时，结账找零易错，且新人培训成本极高。
- **架构演进目标**：实现对标业界前沿的“傻瓜式”防错结算（一键账单拆合），以及高度可视化、无缝数据同步的房态矩阵（Room Rack）。用极致的视觉降噪，降低员工长倒班期间的认知负荷。

## 2. 关键数据资产层级 (Data Hierarchy)
*说明：在此业务域中，哪些数据最重要，必须占据视觉 C 位。*
- **T0 (绝对焦点)**：实时结算总金额 / 待退款押金 / 支付状态。—— 必须占据视觉 F 型动线的最终落点，使用最大字号与主色 A（Neon Cyan: `#00F0FF`）进行高亮呈现。
- **T1 (核心指标)**：当前房间状态（空闲、入住、打扫、维修）/ 核心账单明细项。—— 建议使用色块矩阵（Bento Grid 的独立区块）展示，通过极简的色彩和透明度区分状态。
- **T2 (辅助元数据)**：旅客身份证尾号 / 车辆牌照 / 预订渠道（OTA来源）/ 员工操作工号。—— 必须进行深度的视觉降噪（如 `opacity-40` 或极细无衬线字体），仅作为存根展示。

## 3. UI 架构与视觉映射规范 (Architecture & Visual Mapping)
*说明：业务需求必须严格遵循以下具体的赛博极简前端实现逻辑。*
1. **[防错结算与数字排版]**：数字必须具备最高层级的视觉权重。所有涉及金额（房费、押金、客房服务）的账单列表，强制使用等宽字体（Monospace），并在 CSS 中严格设定 `font-variant-numeric: tabular-nums`，确保小数点和数字绝对垂直对齐。
2. **[房态免读与状态映射]**：房间状态必须脱离文字依赖。运用 Glassmorphism 不透明度控制：空房状态使用 `bg-white/5`，已入住使用 `bg-white/20` 配合发光边框；打扫/维修中等过渡状态，必须呈现轻微的脉冲呼吸动效（Pulse animation）以提示状态变更。
3. **[黄金交互区隔离法则]**：“退房结账”、“一键发票”等高频/高危操作按钮，必须置于 F 型视觉动线的黄金交互区（如独立的右侧控制栏或底部悬浮条）。必须带有明确的 Hover 浮动光影反馈，操作确认感必须极强。

## 4. 绝对反模式 (Anti-Patterns / DO NOT DO)
*说明：在该业务领域中，绝对不允许出现的 UI/UX 设计。*
- ❌ **禁忌一：深层弹窗 (Modal Hell)**：绝对禁止使用多层嵌套的弹窗来办理入住或结算。所有核心流程（选房、刷卡、出单）必须在平铺的栅格面板或单一的右侧滑出抽屉（Off-canvas drawer）中流畅完成。
- ❌ **禁忌二：表单式堆砌**：禁止将旅客信息像传统 Excel 表单一样密密麻麻堆砌。必须采用组件化的“旅客卡片（Guest Card）”进行信息封装。
- ❌ **禁忌三：常规状态使用危险色**：严禁使用红色（Neon Pink）来表示“已入住”等正常运转状态。高亮红色仅限用于“未结清账单”、“房卡挂失”或“超时未退房”等异常警告。
